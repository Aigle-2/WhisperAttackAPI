# ADR-0005: No redistribution of VAICOM-derived data; transparent local generation

**Status:** Accepted
**Date:** 2026-06-18
**Deciders:** Project owner (Aigle_2)

## Context

The repo currently commits `stt_backends/vaicom_keyterms.txt`, which is **derived
from a local VAICOM install** (now generated by the packaged `vaicom_generator_core`
from VAICOM's `keywords.txt`, profiles, and caches). Shipping that file in an
OSS release means **redistributing data derived from VAICOM**, whose license terms
are not confirmed — a licensing grey zone.

Owner's constraint: shipping only the generator is acceptable **only if it stays
transparent for end users** — no manual "wire VAICOM into VAIVOX" step.

## Decision

**Do not ship VAICOM-derived vocabulary.** Ship the **generator** and make
generation transparent:

1. **Auto-discover** the VAICOM install: common Steam/standalone locations, the
   VoiceAttack `Apps\VAICOM*` folder, and the existing `VAICOMPRO_DIR` env override.
2. **Auto-generate on demand**: on first run, and when the generated vocabulary is
   missing or stale, generate it in the background into `%LOCALAPPDATA%\VAIVOX`.
3. **Ship a generic seed that is NOT VAICOM-derived** (phonetic alphabet, widely
   documented DCS callsigns and ATC vocabulary) so VAIVOX works out-of-the-box even
   before the first generation completes.
4. **Expose status + control in the UI**: a clear "VAICOM vocabulary: generated /
   not found" indicator, a "Refresh VAICOM vocabulary" action, and an optional path
   override for non-standard installs.

The same generation step produces the **Axis B phrase index** (ADR for snap to
follow), so this path is reused, not duplicated.

Mission/server F10 menu items are a separate lifecycle: VAICOM imports them as
`Action ...` commands from the live DCS F10 menu and logs them in `VAICOMPRO.log`.
VAIVOX treats those as a non-persistent mission overlay that is polled and hot-applied
at runtime (ADR-0009), not as part of the permanent generated keyterm/phrase-index
files.

After ADR-0012, the mission overlay is a catalog of ephemeral command surfaces rather
than permanent vocabulary. The UI and STT keyterms use the human label (for example
`FLEX NORTH`), while routing preserves VAICOM's identifier (`Action FLEX NORTH`),
`ActionIndex`, and `Command ID` as a typed `VaicomF10Action` target.

## Options Considered

### Option A: Generator-only + auto-discovery (chosen)
**Pros:** licensing-clean; vocabulary always matches the user's *actual* install;
transparent (no manual config in the common case).
**Cons:** needs robust auto-discovery; first-run generation latency.

### Option B: Keep committing the derived list
**Pros:** zero first-run latency.
**Cons:** redistributes VAICOM-derived data under unclear terms; goes stale vs the
user's install.

### Option C: Generator-only but manual path entry
**Cons:** fails the owner's transparency constraint.

## Trade-off Analysis

Option A is the only one that satisfies both "license-clean" and "transparent". The
first-run latency (the only real downside) is masked by the generic seed (step 3)
plus background generation (step 2), so the user is never blocked.

## Consequences

- Easier: no licensing exposure; vocabulary is fresh per install.
- Harder: auto-discovery must handle Steam, standalone, and custom installs; the UI
  must surface generation status clearly.
- Remove `vaicom_keyterms.txt` from version control; document the generic seed's
  provenance to keep it demonstrably non-VAICOM-derived.
